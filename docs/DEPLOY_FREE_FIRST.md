# Free-First Deployment Guide

This repo can be deployed for a real soft launch with the lowest possible cost, but there are important limitations.

## Recommended setup

### Option A: Lowest-friction soft launch

- Frontend: Vercel Hobby
- Backend API: Render Free Web Service
- PostgreSQL: Render Free Postgres
- Redis: Upstash Free Redis
- Storage: S3-compatible bucket later, or leave document file storage features limited at first

This is the easiest path if you want to get a public URL online quickly.

### Option B: Safer if you want to avoid Vercel Hobby restrictions

- Frontend: Render Static Site
- Backend API: Render Free Web Service
- PostgreSQL: Render Free Postgres
- Redis: Upstash Free Redis

Use this if you want both frontend and backend on Render while staying cheap.

## Important limitations

### Render Free

- Free web services spin down after 15 minutes of inactivity
- Free Postgres expires 30 days after creation
- Free Postgres has 1 GB storage and no backups

Source:
- https://render.com/free
- https://render.com/docs/web-services
- https://render.com/docs/postgresql-refresh

### Vercel Hobby

- Free
- Good for the React frontend
- Hobby is documented for personal, non-commercial use

Source:
- https://vercel.com/docs/plans/hobby

### Railway

- Not truly ongoing free tier for production
- Current entry path is a free trial with credits, then paid usage

Source:
- https://railway.com/pricing

### Upstash Redis

- One free Redis database
- 256 MB storage
- 500K commands per month

Source:
- https://upstash.com/pricing/redis

## What is already prepared in this repo

- `frontend/vercel.json`
  - Rewrites all routes to `index.html` so React Router works on Vercel
- `render.yaml`
  - Blueprint for backend API + free Render Postgres

## Deployment steps

### 1. Deploy the backend on Render

1. Push this repo to GitHub
2. In Render, create a new Blueprint deploy and point it at the repo
3. Render will read `render.yaml`
4. After the backend service and Postgres are created, set the missing environment variables manually

Required on Render before production-like use:

- `REDIS_URL`
  - Recommended: use Upstash Redis
- `FRONTEND_URLS`
  - Example:
  - `https://your-frontend.vercel.app`
- `JWT_SECRET`
  - Auto-generated by `render.yaml`, but you can replace it with your own long secret

Optional now, useful later:

- `S3_BUCKET`
- `S3_REGION`
- `S3_ENDPOINT`
- `AWS_ACCESS_KEY_ID`
- `AWS_SECRET_ACCESS_KEY`
- `SMTP_HOST`
- `SMTP_PORT`
- `SMTP_SECURE`
- `SMTP_USER`
- `SMTP_PASS`
- `RD_CLIENT_ID`
- `RD_CLIENT_SECRET`

Recommended initial value:

- `RD_ENVIRONMENT=sandbox`

### 2. Run Prisma migration on Render Postgres

Render will create the database, but you still need schema migration.

Recommended one-time command from your local machine:

```bash
cd backend
DATABASE_URL="paste-render-postgres-url-here" npx prisma migrate deploy
```

If needed, seed once:

```bash
cd backend
DATABASE_URL="paste-render-postgres-url-here" npm run db:seed
```

### 3. Create free Redis on Upstash

1. Create one Redis database on Upstash
2. Copy the `REDIS_URL`
3. Paste it into Render backend env

### 4. Deploy the frontend

#### Vercel path

1. Import the `frontend` directory as a Vercel project
2. Framework preset: `Vite`
3. Set:

```bash
VITE_APP_ORIGIN=https://your-frontend.vercel.app
VITE_OPS_ORIGIN=https://your-frontend.vercel.app
VITE_APEX_ORIGIN=https://your-frontend.vercel.app
```

If the frontend expects an API origin variable in your hosting setup, point it to the Render API URL.

Then update Render:

```bash
FRONTEND_URLS=https://your-frontend.vercel.app
```

#### Render Static Site path

If you don't want to use Vercel Hobby, deploy `frontend` as a static site on Render instead.

## Cheapest real-world recommendation

If you want the cheapest realistic path without painting yourself into a corner:

- Start with:
  - Frontend on Vercel or Render Static Site
  - Backend on Render Free
  - Postgres on Render Free
  - Redis on Upstash Free
  - `RD_ENVIRONMENT=sandbox`
- Upgrade first when needed:
  1. Render Postgres
  2. Render backend web service
  3. Add S3-compatible storage
  4. Add custom domain

## Will future changes be hard?

No, not if you keep this split:

- frontend separate from backend
- database separate from app
- Redis externalized
- secrets in env vars only

That means later you can:

- move frontend from Vercel to Render or Cloudflare Pages
- move backend from Render to Railway/Fly/VM
- move Postgres to managed paid Postgres
- add S3 and SMTP without changing app architecture

## What I would do today

For a real public soft launch with almost no spend:

1. Backend on Render Free
2. Postgres on Render Free
3. Redis on Upstash Free
4. Frontend on Render Static Site if you want fewer policy concerns
5. Keep sandbox RD first
6. Upgrade Postgres before anything else if usage starts to grow
